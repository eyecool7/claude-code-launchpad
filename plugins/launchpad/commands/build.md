# 프로젝트 구축 킥오프

프로젝트 계획서를 읽고, 현재 코드 상태를 분석하여, 이번 세션의 작업 계획을 수립한다.
매 세션 시작 시 실행한다. 몇 번 실행해도 동일한 결과를 보장한다 (멱등).

**전제 조건:** `/launchpad:setup` 완료 상태.

## Step 1: 세팅 검증

아래 파일 존재를 확인한다. 하나라도 없으면 즉시 중단하고 안내한다:

| 파일 | 없을 때 안내 |
|------|------------|
| `project-plan.md` | "계획서가 없습니다. `/launchpad:plan`부터 시작하세요." |
| `CLAUDE.md` | "세팅이 완료되지 않았습니다. `/launchpad:setup`을 실행하세요." |
| `.claude/decisions.md` | "decisions.md가 없습니다. `/launchpad:setup`을 실행하세요." |
| `.claude/lessons.md` | "lessons.md가 없습니다. `/launchpad:setup`을 실행하세요." |

## Step 2: 컨텍스트 로드

아래 파일을 **순서대로** 읽는다:

1. `project-plan.md` 전체 (섹션 1–7. 섹션 7은 setup이 생성한 셋업 결과)
2. `.claude/decisions.md` — 이전 세션의 기술 결정
3. `.claude/lessons.md` — 이전 세션의 교훈

읽은 후 핵심 정보를 내부적으로 정리한다:
- 프로젝트 이름, 스택, 핵심 기능 (섹션 1)
- 에이전트 구조 + 스킬 목록 (섹션 4)
- Phase 목록 + 각 Phase의 완료 기준 + 에이전트·스킬 매핑 (섹션 6)
- 셋업 결과: Tier + 판정 근거, 생성 파일, 호환성 주의사항 (섹션 7)
- 기존 기술 결정 (decisions.md)
- 기존 교훈 (lessons.md)

## Step 3: 코드베이스 상태 스캔

Bash 도구로 아래를 실행하여 현재 상태를 파악한다:

```bash
git log --oneline -20          # 최근 커밋 히스토리
git status                     # 현재 변경사항
ls -R src/ 2>/dev/null | head -60   # src/ 디렉토리 구조
ls package.json tsconfig.json next.config.* .env.local 2>/dev/null  # 설정 파일 존재
```

## Step 4: Phase 판별

`project-plan.md` 섹션 6의 **각 Phase 완료 기준**을 Step 3에서 파악한 코드베이스 상태와 대조한다.

**판별 기호:**
- ✅ 완료: 완료 기준이 전부 충족
- 🔨 진행 중: 완료 기준이 일부만 충족
- ⬜ 미시작: 완료 기준이 하나도 미충족 (선행 Phase 완료 상태)
- 🔒 진행 불가: 선행 Phase가 미완료

**현재 Phase** = 첫 번째 🔨 또는 ⬜ Phase.

## Step 5: Ralph Loop 적합도 판정

현재 Phase(또는 Phase 내 개별 작업)에 대해 Ralph Loop 자동화 적용 가능 여부를 판정한다.

**판정 기준:**
- ✅ 적합: 완료 기준이 **기계적으로 검증 가능** (빌드 성공, 테스트 통과, API 응답 수신, 파일 존재)
- ❌ 부적합: **주관적 판단 필요** (디자인 품질, UX 흐름, 레이아웃 선택)

**일반적 패턴:**
- 프로젝트 초기 세팅, 폴더 구조 생성, API 클라이언트 구현 → ✅
- 파서, 유틸리티 함수, API 엔드포인트 → ✅
- UI 컴포넌트 디자인, 애니메이션, 색상 체계 → ❌
- OAuth 플로우, 배포 설정 → ❌ (사람 확인 필요)

## Step 5.5: 미매핑 스킬 자동 검색

`.claude/skills/` 디렉토리의 모든 SKILL.md를 스캔하여:

1. 현재 Phase의 작업 내용(UI 컴포넌트, 스타일링, API, 데이터 처리 등)과 각 스킬의 `description`을 대조
2. 섹션 6에 매핑되지 않았지만 현재 Phase 작업과 관련성이 높은 스킬을 식별
3. 식별된 스킬이 있으면 Step 6 출력의 **주의사항**에 추가:
   `⚠️ 미매핑 스킬 발견: {스킬명} — {description 요약}. 현재 Phase 작업과 관련성이 높으므로 활용을 권장합니다.`

**매칭 기준 예시:**
- UI 컴포넌트/페이지 작업 → `design-rules`, `ui-ux-pro-max`, `frontend-design`
- Remotion 렌더링 작업 → `remotion`
- PDF 생성 작업 → `pdf`
- WordPress 연동 작업 → `wp-rest-api`
- 보안 관련 작업 → `security-review`
- 리팩토링 작업 → `easy-refactoring`

## 🔧 이번 세션 구축 순서

현재 Phase의 미달성 항목 중 **2–4개**를 선정하여, 섹션 6에 기록된 에이전트·스킬·파일 위치 정보를 포함한 구축 순서를 제시한다.

| # | 작업 | 담당 에이전트 | 활용 스킬 | TDD | 완료 기준 |
|---|------|:----------:|----------|:---:|----------|
| 1 | {작업명} ({파일 위치}) | {에이전트명 또는 "—"} | {스킬명 또는 "—"} | {✅/❌} | {검증 가능한 완료 조건} |

## Step 7: 프로세스 리마인드

출력 마지막에 아래 3줄을 반드시 포함한다:

> **📝 기록 규칙 (매 세션 리마인드)**
> 1. **decisions.md**: 기술 결정 발생 시 즉시 기록. 코딩 전에 기록한다. "나중에"는 실패한다.
> 2. **lessons.md**: 빌드/테스트 실패 해결 후 즉시 기록. 같은 실수 반복 방지.
> 3. **세션 종료 전**: "계획서 대비 달라진 점이 있는가?" 자체 점검. 미기록 결정 소급 기록.
> 4. **TDD**: ✅ 작업은 테스트를 먼저 작성한 후 구현한다. 실패하는 테스트 → 구현 → 통과.

## Step 8: 구축 실행 (승인 후)

사용자가 승인하면 CLAUDE.md "개발 프로세스" 섹션의 Tier에 따라 구축을 실행한다.

### 공통 규칙 (모든 Tier)

- **스킬 활용**: Step 6 테이블의 "활용 스킬" 열에 스킬명이 있으면 해당 작업 시작 전 Skill 도구로 invoke하여 지침을 로드한다. Step 5.5에서 미매핑 스킬이 발견된 경우, 해당 작업에서 그 스킬도 함께 invoke한다.
- **TDD**: ✅ 작업은 테스트를 먼저 작성한 후 구현한다. 실패하는 테스트 → 구현 → 통과.
- **검증**: 각 작업 완료 시 CLAUDE.md의 검증 명령을 실행한다. 실패 시 즉시 수정하고 재검증한다. 2회 연속 실패하면 사용자에게 보고하고 중단한다.
- **진행 관리**: TodoWrite로 작업 상태를 추적한다. 완료 즉시 completed 처리.
- **기록**: 기술 결정 발생 시 decisions.md, 빌드/테스트 실패 해결 시 lessons.md에 즉시 기록.

### Ralph Loop 연동

Step 5에서 **Ralph Loop ✅**로 판정된 작업은 사용자 확인 없이 연속 실행한다:
- ✅ 작업: 구현 → 검증 명령 통과 확인 → 다음 작업으로 자동 진행. 중간에 사용자에게 확인을 묻지 않는다.
- ❌ 작업: 구현 완료 후 사용자에게 결과를 보여주고 확인을 받은 뒤 다음 작업으로 진행한다.

✅ 작업끼리 연속으로 이어지면 검증만 통과하면 멈추지 않고 계속 진행한다. ❌ 작업을 만나면 거기서 멈추고 사용자 피드백을 받는다.

### Tier 1 — 순차 실행

Step 6 테이블 순서대로 하나씩 구현한다. 메인 컨텍스트에서 직접 코딩.

### Tier 2 — 서브에이전트 위임

1. Step 6 테이블에서 **"담당 에이전트"가 있는 작업**을 식별한다.
2. **병렬 그룹** 판정에 따라 독립 작업은 Agent 도구를 **단일 메시지에서 동시 호출**한다.
3. Agent 도구 호출 시:
   - `subagent_type`: Step 6의 에이전트명 (`.claude/agents/{에이전트명}.md`에 정의됨)
   - 프롬프트에 포함할 것:
     - 작업명 + 대상 파일 경로
     - 완료 기준 (검증 가능한 조건)
     - 활용 스킬이 있으면: "작업 시작 전 Skill 도구로 {스킬명}을 invoke할 것"
     - TDD ✅이면: "테스트를 먼저 작성한 후 구현할 것"
     - decisions.md에서 현재 작업과 관련된 기술 결정 (있으면)
4. **"담당 에이전트"가 없는 작업** (— 표시)은 메인에서 직접 구현한다.
5. 모든 에이전트 결과 수신 후 검증 명령을 실행한다.

### Tier 3 — 에이전트 팀

Tier 2와 동일하되, Agent 도구 호출 시 `isolation: "worktree"`를 사용하여 각 에이전트가 격리된 환경에서 작업한다.

---

## 주의사항

- Step 1–7은 **읽기 전용**이다. 어떤 파일도 생성·수정하지 않는다.
- Step 8은 사용자 승인 후 **코드 생성·수정**을 수행한다.
- 세션 목표는 **2–4개**로 제한한다. 컨텍스트 한계를 고려하여 한 세션에 Phase 전체를 완료하려 하지 않는다.
- Phase 판별은 코드베이스 상태에서 유도한다. 별도 상태 파일을 사용하지 않는다.
- 구축 순서를 출력한 후 사용자의 승인을 기다린다. 승인 전에 코딩을 시작하지 않는다.
- 구축 순서의 에이전트·스킬 정보는 섹션 6에 기록된 것을 그대로 표시한다. 독자적으로 매핑을 생성하지 않는다.
